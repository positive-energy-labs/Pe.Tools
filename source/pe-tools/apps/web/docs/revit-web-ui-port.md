# Revit web-UI port â€” learnings log

Porting two routes from the old `signalir-clientside-demo/v3` into `apps/web`:
`family-matrix` (read-only matrix over loaded families) and `settings-prototype`
(schema-driven settings editor). Treated as an exercise: **can we reasonably
build web UIs over Revit code?** The demo predates many locked decisions, so it
is a source of _patterns_, not gospel.

This doc records what the codebase got right/wrong for a web-UI-heavy future,
written at each checkpoint.

---

## Checkpoint 0 â€” Recon & architecture decisions

### What the demo had

- `src/lib/host/*` â€” bespoke HTTP client (`SettingsHostHttpClient`) hardcoding
  `localhost:5180` + `HttpRoutes`, an SSE `EventSource` client, TanStack Query
  hooks, a query-key factory, a `@legendapp/state` route store, and a
  multi-session selector (`useHostSessionSelection`).
- `packages/schema-core/src/*` â€” pure-TS JSON-schema engine: resolves
  `$ref`/`allOf`/`anyOf`/`oneOf`/`if-then-else` into an "effective node",
  computes defaults, diffs, `SchemaDocument`/`SchemaNodeRef` API. **No deps.**
- `src/lib/schema-to-field-render/*` â€” renders a `SchemaDocument` + TanStack Form
  into shadcn fields (scalar/object/array/table), with remote field-options.
- ~150 hand-free TypeGen-generated TS types/enums in `src/generated/`.
- **No validation anywhere** â€” responses were cast (`as TResponse`).

### What the new app (`apps/web`) already has

- `@pe/host-generated/contracts` â€” a **generated operation catalog**
  (`hostOperations`) with `key`, `verb`, `route`, `requestShape`/`responseShape`
  (top-level field descriptors only), examples, cost/visibility metadata.
  Generated by the **C# CLI** `pe-dev codegen sync --target host-contracts`.
- `@pe/host-generated/types` â€” named TS interfaces/enums, but only for a
  **curated subset** (scripting, context, host-probe/session-summary). The
  loaded-families + settings payload types are **absent**.
- `@pe/host-client` â€” Node-only client (`PeHostClient` spawns the host via
  `child_process`/`fs`). Already establishes the **zod-at-boundary** house
  pattern: `z.object({...}).strip()` schemas typed as `z.ZodType<Iface>` +
  `isSchema()` guard. Not usable from the browser.
- `routes/ops.tsx` â€” the canonical **browser** caller: `fetch` through the
  `/pe-host` Vite proxy (`vite.config.ts` strips the prefix â†’ `localhost:5180`).
- Design system: `@base-ui/react` (v1.6), shadcn style `base-mira`, lucide
  icons, Tailwind v4, `#/*` import alias. Existing `ui/`: button, command,
  dialog, dropdown-menu, input, input-group, textarea.

### Operations the routes need (all exist in the catalog)

| Route    | Operation key                            | verb | route                                |
| -------- | ---------------------------------------- | ---- | ------------------------------------ |
| both     | `settings.host-probe`                    | GET  | `/api/settings/host-probe`           |
| both     | `settings.session-summary`               | GET  | `/api/settings/session-summary`      |
| matrix   | `revit.catalog.loaded-families`          | POST | `/api/revit/catalog/loaded-families` |
| matrix   | `revit.matrix.loaded-families`           | POST | `/api/revit/matrix/loaded-families`  |
| settings | `settings.workspaces`                    | POST | `/api/settings/workspaces`           |
| settings | `settings.tree`                          | POST | `/api/settings/tree`                 |
| settings | `settings.schema`                        | POST | `/api/settings/schema`               |
| settings | `settings.field-options`                 | POST | `/api/settings/field-options`        |
| settings | `settings.parameter-catalog`             | POST | `/api/settings/parameter-catalog`    |
| settings | `settings.document.{open,validate,save}` | POST | `/api/settings/document/*`           |

### Decisions (with rationale)

1. **Browser host caller over `/pe-host`, not `@pe/host-client`.** The Node
   client can't run in the browser. Generalize `ops.tsx`'s `callOp` into a small
   `#/host/client` that takes a `HostOperationDefinition` from the generated
   catalog + a zod schema and returns validated data. â†’ _leverages generated
   routes; validates at the real network boundary._

2. **Hand-author loose zod contracts in the web app, infer types from them.**
   The payload types aren't in `@pe/host-generated`. The user wants zod at
   semi-fragile boundaries; the HTTP response is exactly that (the demo skipped
   it). Schemas kept loose/nullish (`.partial()`, `.nullish()`, `.passthrough()`
   where unknown) per the existing zod-boundary guidance. Shapes mirror the
   demo's TypeGen output, which is faithful to the C# contracts (verified).

3. **Drop the multi-session layer.** The new host has **no session list**:
   `host-probe` gives bridge status, `session-summary` gives a _single_ active
   session, and the loaded-families requests take **no `target`/session
   selector**. So `useHostSessionSelection`/`BridgeSessionSelector` are not
   ported â€” replaced by a connection/active-document status strip. Big
   simplification the demo's design didn't anticipate.

4. **SSE is degradable.** The operation catalog has no events endpoint; whether
   Pe.Host exposes `/api/settings/events` is unverified. The demo used SSE only
   for toast notifications (matrix) and stale-invalidation (settings) â€” both
   non-critical. Port as best-effort: if the endpoint 404s, the UI just doesn't
   show live updates. **Open question:** does the new host expose SSE?

5. **Copy `@pe/schema-core` + `schema-to-field-render` close to verbatim.**
   These are the crown jewels (user-flagged "absolutely critical"). They're
   internal pure logic + UI, not boundaries, so no zod added. Only import paths
   and a few generated-type references get rewired.

6. **Reimplement the settings route store without `@legendapp/state`.** Pulling
   a new state lib for one route fails the ponytail ladder; the store is route
   glue, not a lib. Target a minimal `useSyncExternalStore` store (matches the
   codebase's store philosophy; TanStack DB already rejected here). `@tanstack/
react-form` _is_ required (schema-render depends on it) â€” that's a real add.

### The codegen takeaway (what's right / wrong for the future)

- **Right:** the C# contracts already carry `[ExportTsInterface]`/`[ExportTsEnum]`
  (same TypeGen that made the demo's `src/generated/`). The shapes are stable
  and generated â€” low regression risk.
- **Wrong / gap:** `pe-dev codegen --target host-contracts` only emits the
  operation catalog + a curated type subset, so consumers of _data payloads_
  (these routes) have no generated types to import. The hand-authored zod
  schemas in `#/host/contracts` are effectively the **spec for a future codegen
  target** â€” e.g. `--target web-contracts` that emits every `[ExportTsInterface]`
  type _and_ a zod schema beside it. That closes the loop the user wants
  ("leverage generated types" + "zod at boundaries") but is C# tooling work
  outside this JS port.

### Test philosophy applied

Per the user: end state has few/no unit tests. The pure schema engine
(`@pe/schema-core`) is the one fragile, regression-prone, easily-unit-tested
boundary â€” it keeps the demo's tests if any, else gets one small graph-level
check. Everything else is verified by running the routes against a host.

## Checkpoint 1 â€” family-matrix ported

Shipped the whole wiring layer + the read-only matrix route.

### Files added (`apps/web/src/`)

- `host/contracts.ts` â€” loose zod schemas + inferred types for the
  loaded-families payloads, host-probe, session-summary. Enums modelled as const
  objects + `z.enum(obj).catch(safeMember)` so a stray host value degrades
  instead of throwing on a big matrix. Objects `.loose()` (tolerate new fields),
  optional fields `.nullish()`, arrays `.default([])`.
- `host/client.ts` â€” `callHostOp(key, schema, request?)`: fetch the generated
  op's verb+route through `/pe-host`, then `schema.safeParse` the body. Throws
  `HostCallError` on non-2xx or shape mismatch. Generalized from `ops.tsx`.
- `host/queries.ts` â€” TanStack Query hooks (`useHostProbeQuery`,
  `useSessionSummaryQuery`, `useLoadedFamiliesCatalogQuery`,
  `useLoadedFamiliesMatrixQuery`). No session target; query keys derive from the
  filter.
- `components/ui/select.tsx` â€” base-ui Select wrapper (demo's was radix), generic
  over value, styled to match `dropdown-menu.tsx`.
- `components/ui/tooltip.tsx` â€” base-ui, copied verbatim from the demo.
- `routes/family-matrix.tsx` â€” cleanroom: the three table components
  (`MasterSpreadsheetTable`, `FamilyParameterTable`, `FamilyCard`) are nearly
  verbatim (they consume the identically-shaped contract types); the container
  was rewritten to drop session selection + SSE and gate on `bridgeIsConnected`,
  with a connection/active-doc status pill.

### What this validated about the foundation

- **The generated op catalog is enough to drive real calls from the browser.**
  `callHostOp` needs nothing but `hostOperations[key]` + a zod schema. Good.
- **The zod boundary is cheap to add and the right call** â€” the matrix payload is
  deeply nested and host-controlled; `.loose()/.catch()` keeps it resilient
  without per-field ceremony. This is the pattern a codegen target should emit.
- **Type-aware `vp check` is strong verification for a port**: it validates every
  base-ui subcomponent/prop and the generated-catalog keys. Passed clean (44
  files, 0 errors/warnings). `routeTree.gen.ts` flags a biome format issue but
  that file already violated format on HEAD (generated; `--fix` won't rewrite it).

### Open questions / cut corners

- **Live-host render unverified.** No Pe.Host on `localhost:5180` in this session
  and the dev server (`pe-dev web pea`, port 5173) is owned by another process,
  so the matrix wasn't exercised against real data. The disconnected state is
  correct by construction. **Action:** open `/family-matrix` against a running
  Pe.Host + Revit to confirm payload shapes parse.
- **Filter exactness.** C# `LoadedFamiliesFilter` confirmed to accept
  `familyNames`/`categoryNames`/`placementScope`, so exact family selection is
  sent server-side (not client-filtered). If `strictRequestValidation` rejects
  anything, it surfaces as a query error in the UI.
- **SSE notifications dropped** (see decision 4). The demo's live host-message
  toast panel is gone until SSE on the new host is confirmed.

## Checkpoint 2 â€” settings-prototype ported (schema engine + store)

The harder route: schema-driven form over the full open/validate/save loop.

### Files added/copied (`apps/web/src/`)

- `packages/schema-core/src/**` â€” copied **verbatim** from the demo. Only edit:
  `core/types.ts` now imports the `x-ui`/`x-options` type-deps from generated Host contracts.
- `lib/schema-to-field-render/**` â€” copied, then rewired: imports point at
  `#/components/ui/*`, `#/host/queries`, `#/host/settings-contracts`. The
  session layer was removed â€” `sessionId` â†’ `rootKey` throughout (the new
  `settings.field-options` endpoint requires `rootKey` and takes no session
  `target`).
- `host/settings-contracts.ts` â€” loose zod for every settings payload
  (workspaces, tree, schema, field-options, parameter-catalog, document
  snapshot/save, validation) + request interfaces + the two schema-extension
  type-deps the engine needs. `SettingsDirectoryNode` is recursive â†’ `z.lazy`.
  `parameterCatalogEntry` mirrors `identity.name` onto a flat `name` the render
  lib reads.
- `host/settings-store.ts` â€” the route store, reimplemented on
  `useSyncExternalStore` (no `@legendapp/state`). Holds selection + the
  document/validation/save state machine; derives one frozen snapshot per
  mutation; actions call the host via `callHostOp`.
- `host/queries.ts` â€” extended with `useWorkspaces/Tree/Schema/FieldOptions/
ParameterCatalog` hooks.
- `components/ui/{combobox,switch,label}.tsx` â€” combobox copied (already base-ui,
  icons swapped remixâ†’lucide); switch rewritten for base-ui; label is a native
  `<label>` (base-ui has no Label primitive).
- `routes/settings-prototype.tsx` â€” **leaner cleanroom**: connection pill +
  workspace/module/root/file selects + schema form + validation/save/deps/
  composed panels. The demo's verbose debug popover & state dumps were dropped.
- `packages/schema-core/tests/meta-schema-core.test.ts` â€” one graph-level test (the fragile boundary):
  `$ref` resolution, `allOf` merge, default synthesis. Passes.

### What this validated about the foundation

- **The schema engine ported with zero logic changes** â€” `$ref`/`allOf`/defaults
  all work against the new app (test-confirmed). It being dependency-free is why
  the copy was trivial. This is the codebase's strongest reusable asset.
- **The render lib needed only boundary edits** (UI import paths + the
  sessionâ†’rootKey swap). Its internals (TanStack Form binding, field-options
  resolution, custom table renderer) are intact. Good separation.
- **`useSyncExternalStore` fully replaced legendapp** for this route's needs
  without granular observables â€” a single whole-snapshot read is plenty. No
  reason to add a state lib.
- **base-ui vs radix is the main UI-porting cost**: select/switch/label/combobox
  all differ. `asChild` (radix) â†’ `render={<Comp/>}` (base-ui) is the recurring
  gotcha.

### Cut corners / open questions

- **`@tanstack/react-form` â†” `SettingsForm` cast.** The lib defines a structural
  `SettingsForm`; the real form api is a superset, so the route casts at the
  `SchemaToFieldRender` boundary. Faithful to how the demo coupled them.
- **Field-options `rootKey`** is now threaded, but **unverified against a live
  host** â€” confirm the dynamic dropdowns resolve.
- **Store `validateDraft`/`markStale`/`clearStale`** exist but aren't wired to UI
  yet (validate-on-save covers the common path; stale needs SSE).
- Same live-host caveat as Checkpoint 1: shapes parse by construction but weren't
  exercised against real settings data.

### Net answer to "can we reasonably build web UIs over Revit code?"

Yes, and the cost is mostly at the **boundary**, not the core. The generated
op catalog + a thin browser caller + loose zod gave both routes real host wiring
in a few hundred lines; the schema engine and render lib moved over almost
untouched. The biggest leverage win available is a `pe-dev codegen` target that
emits the payload types + zod so `#/host/*contracts.ts` stops being hand-written.

## Checkpoint 3 â€” live debugging against a real host (family-matrix)

Ran the route against a real Pe.Host + Revit 2025 model. Three real bugs, all
now fixed and verified end-to-end (loaded a 3-family Sprinklers matrix: 3
families / 7 types / 34 parameters rendering correctly).

### Bug 1 â€” the matrix DTO had drifted from the demo's shape (the "error")

The host's matrix `visibleParameters`/`excludedParameters` no longer match the
demo's old TypeGen types my contract mirrored:

- parameter fields (`identity`, `isInstance`, `dataType*`, `groupType*`) are now
  nested under **`definition`** (were flat);
- **`scope` was renamed to `presence`** (same enum members);
- `valuesByType` cells can be **`null`**.

My `z.record(z.string(), z.string())` threw on the null cells â†’ `safeParse`
failed â†’ `HostCallError` â†’ the matrix query errored. Fixed in `contracts.ts`
with a `.transform()` that flattens `definition`, maps `presence`â†’`scope`, and
coerces nullâ†’"" â€” so the proven render code stays untouched. Guarded by a fixture
test (`tests/loaded-families-contract.test.ts`) built from the captured real JSON.
**Lesson: the hand-authored contracts drift from the C# source silently. This is
the strongest argument for the codegen target â€” a generated zod schema would have
moved with the contract.**

### Bug 2 â€” catalog budget capped the pickers at 50

The catalog defaults to `maxEntries: 50`. The category picker is derived from an
all-families catalog call, so it only ever saw the categories present in the
first 50 of 582 families (4 categories). Selecting a category then capped its
family list at 50 too. Fixed by passing a generous `budget.maxEntries` on the
(cheap, parameter-free) catalog calls; the expensive matrix budget is sized to
the explicit family selection. After the fix the picker shows all ~66 categories.

### Bug 3 â€” silent failure hid a busy bridge

Pe.Host serializes bridge ops (single-flight). A full Mechanical Equipment matrix
(~7 min) was still running and every other bridge call returned _"Revit bridge is
busy executing 'revit.matrix.loaded-families' for 99153 ms"_. The catalog query
error was swallowed â€” the UI just showed a blank "No categories available", which
made the cause invisible. Fixed by surfacing catalog/family query errors inline.
**This is the connection-state UX gap to keep pulling on:** bridge-busy, bridge-
disconnected, and parse failures should all be legible, and long matrix loads
need progress/cancel affordances (the 7-minute full collection blocks everything).

### Debugging note for future sessions

`window.__TSR_ROUTER__.options.context.queryClient.getQueryCache().getAll()` reads
live TanStack Query state (status/error/data) from the browser â€” the fastest way
to see _why_ a query is empty when the network call itself looks fine.

### Minor / deferred

- Placement-scope `<Select>` shows the raw enum value ("AllLoaded") instead of a
  label â€” cosmetic.
- Settings-prototype route still unexercised against a live host; the same
  matrix-style DTO drift may exist in its parameter-catalog/field-options
  payloads â€” check when wiring it up.

## Checkpoint 3 â€” C# is now the type authority for loaded-families (codegen)

Closed the gap Checkpoint 0/1 flagged: the hand-written loaded-families zod is
gone; the boundary is generated from C#.

### What changed (C# codegen)

- **Un-pruned the types.** Added the loaded-families request/response roots to
  `HostOperationsCatalog.TypeScriptClientExtraTypeRoots`. The reachability prune
  then emits the whole closure into `@pe/host-generated/src/types`.
- **New zod emitter** `Pe.Dev.Cli/Commands/Codegen/HostZodProjection.cs`, wired
  into the `host-contracts` target. It reflects the exported C# types and writes
  `src/zod/host-zod.generated.ts` (exposed as `@pe/host-generated/zod`).
- The emitter reflects C# directly, so it captures nullability TypeGen drops.
  **Proven:** `Dictionary<string, string?>` â†’ interface `{ [k]: string }` but
  schema `z.record(z.string(), z.string().nullable())` â€” and the wire really
  sends null cells. Enums emit as `z.enum(TsEnum)` so a schema's inferred type is
  the nominal enum (value usage + parsing stay unified).

### What changed (web app)

- `host/contracts.ts` is now a thin faÃ§ade: re-exports generated schemas
  (`@pe/host-generated/zod`) + value enums (`@pe/host-generated/types`), and
  holds the ONE hand-maintained piece â€” a `flattenMatrix` view adapter mapping
  the faithful nested wire shape (`definition`/`presence`/null cells) to the flat
  model the table renders. A C# change moves the generated types and the adapter
  fails to compile â†’ drift surfaces at typecheck.
- `host/queries.ts` matrix hook parses with the generated schema, then flattens.
- The hand enum const-objects, hand payload schemas, and `.transform()` deleted.

### Verification (this session)

- `pe-dev codegen sync --target host-contracts` emits the zod file; host-generated
  typechecks clean (81 files).
- Web app typechecks clean (74 files); `vp test` 10/10.
- **Live**: called `revit.matrix.loaded-families` on Pe.Host (5180) with 3
  Mechanical Equipment families (â‰¤3 per the perf rule â€” 50 fams â‰ˆ 8 min) and
  parsed the raw 71 KB response through the generated schema + adapter: 3
  families, all cells string-coerced, null cells â†’ "".

### Cut corners / follow-ups

- **Settings types still hand-zod** (`host/settings-contracts.ts`). Same
  treatment: add their roots to `TypeScriptClientExtraTypeRoots`, regen, swap the
  web imports to `@pe/host-generated/zod`.
- **No opâ†’schema registry yet.** Callers still pass the schema to
  `callHostOp(key, schema, req)`. A generated `Record<HostOperationKey, {request,
response}>` would let `callHostOp(key, req)` infer and drop the arg.
- **`@pe/host-client` (Node) still hand-writes schemas+guards** for its slice;
  now redundant with the generated zod and could import it instead.
- Regen ordering: the generated zod imports enums from `src/types`, so run
  `--target host-types` then `host-contracts` (or `--target all`) after C# edits.

## Checkpoint 4 â€” e2e type+validation safety; all three follow-ups closed

The "ideal shape" pass. C# is now the single authority for the whole public
operation surface, including validation rules.

### Codegen (C#)

- **Full public surface un-pruned.** The prune now roots every
  `HostOperationsCatalog.PublicHttp` request/response type (best-effort â€” a type
  without `[ExportTsInterface]` is skipped, not fatal), on top of the curated
  roots. 306 zod schemas now generated.
- **DataAnnotations â†’ zod refinements.** `HostZodProjection` reads
  `[Range]/[MinLength]/[MaxLength]/[StringLength]/[RegularExpression]/[Required]`
  â†’ `.min()/.max()/.regex()` etc. NJsonSchema reads the same attributes for the
  settings JSON Schema, so **DataAnnotations are the single validation SOT** for
  C# server validation + TS. Proven: `[Range(1,100000)]` on
  `RevitDataOutputBudget.MaxEntries` â†’ `z.number().min(1).max(100000)`.
- **Opâ†’schema registry** (`host-op-schemas.generated.ts`, `@pe/host-generated/
zod/registry`) maps each op key â†’ `{request, response}` schema.
- Recursive types emit zod-v4 getters (`SettingsDirectoryNode`).

### Web (the 3 follow-ups + Ajv)

1. **Registry caller.** `callHostOp(key, request?)` infers the response type from
   the registry and validates request + response â€” no hand-passed schema.
2. **Settings boundary generated.** `settings-contracts.ts` is now a facade over
   `@pe/host-generated/zod` + `/types`; `x-options` and `x-ui` metadata shapes
   are generated from C# shared storage-runtime contracts.
3. **host-client (Node)** dropped ~190 lines of hand schemas/guards for imports
   from `@pe/host-generated/zod`.

- **Ajv client validation** (`@pe/host-client/settings-validation`): the settings editor validates
  the authored doc against the same runtime JSON Schema the host uses
  (server NJsonSchema â†” client Ajv parity), merged into the form's field errors
  for instant feedback. The `x-display-name` label the C# emits is now read by
  the renderer (was a dead input).

### The decided architecture (for future reference)

Two streams, each with the right tool:

- **Operation surface** (build-time, known types): generated zod is the single TS
  projection (`z.infer` for types); validation rules ride C# DataAnnotations.
  Types+schema can't drift (one artifact); drift vs C# surfaces at regen+compile.
- **Authored settings** (runtime, composed schema): the JSON Schema string is the
  contract; both sides validate it (NJsonSchema server / Ajv client). No static
  type is forced onto user-authored documents â€” runtime schema validation is the
  correct safety model there.

`transform`/`json-schema-to-zod` was evaluated and **not** used: it reads the
_generated TS_, which has already lost the dict-value nullability, so it would
re-introduce the null-cell bug. C# reflection is the only layer where that
nullability still exists.
