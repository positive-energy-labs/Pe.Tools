## Core Difference

ACP and AG-UI are both client-facing agent protocols, but they optimize for different clients and therefore preserve different facts.

| Dimension              | ACP                                                                                                         | AG-UI                                                                                                          | Pea adapter consequence                                                                                                                     |
| ---------------------- | ----------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Primary user           | Code editor / IDE / CLI client controlling a coding agent.                                                  | User-facing web/app frontend embedding an agentic backend.                                                     | ACP should feel like an editor subprocess protocol; AG-UI should feel like a web event stream.                                              |
| Main mental model      | `Client` opens `Session`, sends `session/prompt`, receives `session/update`, may answer agent requests.     | App calls `runAgent` with `threadId` / `runId` / messages / state / context, receives typed events.            | ACP session lifecycle is command-heavy; AG-UI lifecycle is event-heavy.                                                                     |
| Transport baseline     | JSON-RPC 2.0, stdio SHOULD be supported; custom transports preserve JSON-RPC lifecycle.                     | Transport-agnostic event stream; HTTP SSE is the common reference path.                                        | Pea exposes ACP stdio and HTTP/SSE-shaped JSON-RPC; AG-UI exposes local HTTP/SSE `/agui/run`.                                               |
| Stream primitive       | `session/update` notification carrying `SessionUpdate` union.                                               | `BaseEvent` union discriminated by `EventType`.                                                                | ACP mapping compresses everything into updates; AG-UI mapping fans events into lifecycle/message/tool/state/custom events.                  |
| Tool representation    | `tool_call` plus `tool_call_update`; has `kind`, `status`, `rawInput`, `rawOutput`, `content`, `locations`. | `TOOL_CALL_START`, `TOOL_CALL_ARGS`, `TOOL_CALL_END`, `TOOL_CALL_RESULT`, plus `CUSTOM`.                       | ACP can keep status and raw input/output in the same card; AG-UI needs lifecycle events plus custom side channels for partial/shell output. |
| Message representation | `agent_message_chunk` / `user_message_chunk` / `agent_thought_chunk` chunks.                                | `TEXT_MESSAGE_START` / `CONTENT` / `END`, plus convenience chunk events.                                       | Pea ACP emits only assistant text chunks; Pea AG-UI emits explicit start/content/end.                                                       |
| Plan representation    | Native `plan` update; entries replace the current plan.                                                     | No plan-specific core event; use `CUSTOM`, `ACTIVITY_*`, or state.                                             | Pea ACP maps tasks/plans into native `plan`; Pea AG-UI currently uses `CUSTOM pea.plan.*`.                                                  |
| State representation   | Session state is mostly lifecycle/session metadata; plans/config/modes are update families.                 | First-class `STATE_SNAPSHOT`, `STATE_DELTA`, `MESSAGES_SNAPSHOT`, `ACTIVITY_*`.                                | Pea AG-UI sends input state/messages snapshots before runtime events; ACP does not have an equivalent state snapshot.                       |
| Human-in-loop          | `session/request_permission` is bidirectional JSON-RPC; permission result returns to agent.                 | `RUN_FINISHED` may carry `outcome: { type: "interrupt", interrupts: [...] }`; next run can include `resume[]`. | Pea ACP has a permission bridge path; Pea AG-UI converts interrupts into run outcome and resume decisions into prompt context.              |
| Run boundary           | Prompt request/response is the boundary; stop reason is returned by `session/prompt`.                       | `RUN_STARTED` / `RUN_FINISHED` / `RUN_ERROR` are event-level facts.                                            | Mastra `run_started` / `run_finished` are not ACP updates; AG-UI wrapper emits native run events.                                           |
| Replay/history         | `session/load` replays conversation via `session/update`; `session/resume` reconnects without replay.       | Pea implements event replay with per-thread sequence via `GET /agui/events`.                                   | Both use Pea protocol-visible history; neither restores native Mastra model memory yet.                                                     |
| Client authority       | Editor may provide cwd, roots, MCP config, permission decisions, filesystem/terminal capability.            | Frontend provides messages, state, context, forwarded props, resume decisions.                                 | Pea keeps runtime authority local; protocols steer context and collect decisions.                                                           |
| Best fit               | IDE-integrated coding agents, editor file context, subprocesses, permission UX, terminal/file features.     | Browser/operator UI, live progress, app state sync, multimodal frontend context, resumable UI streams.         | ACP is better for T3 Code / editor parity; AG-UI is better for native Pea browser/app frontend.                                             |

## Adapter Boundary

Current intended flow:

```text
Mastra HarnessEvent
  -> PeaRuntimeEvent
  -> ACP SessionUpdate
  -> ACP client UI

Mastra HarnessEvent
  -> PeaRuntimeEvent
  -> AG-UI BaseEvent
  -> AG-UI frontend
```

The local invariant is important: protocol adapters must not import `HarnessEvent`. They consume `PeaRuntimeEvent`.

| Boundary              | Current file                         | Contract type                                           | Adapter posture                                                                                                                                                                  |
| --------------------- | ------------------------------------ | ------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Mastra -> Pea runtime | `mastra-harness-runtime-events.ts`   | `HarnessEvent` -> `PeaRuntimeEvent[]`                   | Selective normalization. Drops many UI/session/OM/status events. Sanitizes JSON payloads. De-dupes message start/end. Normalizes subagent inner tools into synthetic tool calls. |
| Pea runtime           | `pea-runtime-events.ts`              | `PeaRuntimeEvent` union                                 | Protocol-neutral event contract. Good audit seam.                                                                                                                                |
| Pea runtime -> ACP    | `acp/pea-runtime-to-acp-events.ts`   | `PeaRuntimeEvent` -> `SessionUpdate[]`                  | Tool/text/plan centric. Drops run boundaries and runtime errors.                                                                                                                 |
| Pea runtime -> AG-UI  | `agui/pea-runtime-to-agui-events.ts` | `PeaRuntimeEvent` -> `BaseEvent[]`                      | Message/tool/custom centric. Drops run boundaries inside translator because wrapper owns run lifecycle.                                                                          |
| AG-UI wrapper events  | `agui/pea-agui-agent.ts`             | `RunAgentInput` -> `RUN_*`, snapshots, sequenced events | Adds `RUN_STARTED`, `STATE_SNAPSHOT`, `MESSAGES_SNAPSHOT`, `RUN_FINISHED`, `RUN_ERROR`, sequence replay.                                                                         |
| Interrupt collection  | `pea-runtime-interrupts.ts`          | `PeaRuntimeEvent` -> run `outcome`                      | AG-UI-only native projection today. ACP permission request path is handled in ACP session/client bridge, not this mapper.                                                        |

## Pea Runtime Event Map

This is the protocol-neutral event surface that ACP and AG-UI actually consume.

| PeaRuntimeEvent                                    | ACP projection                                                                                 | AG-UI projection                                                                                                | Fidelity / adapter depth                                                                                                                                                                               |
| -------------------------------------------------- | ---------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `run_started`                                      | None. ACP prompt request itself starts work.                                                   | Wrapper emits `RUN_STARTED` before runtime send. Translator ignores this event.                                 | Deliberate split: ACP has no run event update; AG-UI run lifecycle is wrapper-owned.                                                                                                                   |
| `run_finished(reason)`                             | None in `SessionUpdate`; ACP `prompt` returns `stopReason`.                                    | Wrapper emits `RUN_FINISHED` with `outcome` and `rawEvent.stopReason`. Translator ignores this event.           | ACP loses event-stream visibility but returns command result; AG-UI preserves run boundary.                                                                                                            |
| `assistant_message_started(messageId)`             | None. First text delta creates visible stream chunk only.                                      | `TEXT_MESSAGE_START { messageId, role: "assistant" }`.                                                          | ACP has no explicit assistant start update in current mapper.                                                                                                                                          |
| `assistant_message_delta(messageId, delta)`        | `agent_message_chunk` with text content.                                                       | Ensures `TEXT_MESSAGE_START`, then emits `TEXT_MESSAGE_CONTENT`.                                                | Strong for text. AG-UI is more lifecycle-complete.                                                                                                                                                     |
| `assistant_message_finished(messageId)`            | None.                                                                                          | `TEXT_MESSAGE_END`.                                                                                             | ACP has no explicit message end update in current mapper.                                                                                                                                              |
| `tool_started(streaming_input)`                    | `tool_call` or `tool_call_update`, status `in_progress`, raw input if present.                 | `TOOL_CALL_START`, optional `TOOL_CALL_ARGS` if input exists.                                                   | Status nuance `streaming_input` collapses to ACP `in_progress`; AG-UI has no status field.                                                                                                             |
| `tool_started(running)`                            | `tool_call`, status `in_progress`, `kind`, title, raw input/content preview.                   | `TOOL_CALL_START`, optional full input as stringified `TOOL_CALL_ARGS`.                                         | Strong lifecycle. ACP preserves kind/status/raw input better.                                                                                                                                          |
| `tool_started(pending_approval)`                   | `tool_call`, status `pending`; ACP session store may request permission through client bridge. | `TOOL_CALL_START` plus args; interrupt collector adds `tool_approval_required` to final `RUN_FINISHED.outcome`. | ACP has native bidirectional permission method; AG-UI has run-finished interrupt/resume model.                                                                                                         |
| `tool_started(suspended)`                          | `tool_call`, status `pending`; no native suspended status.                                     | `TOOL_CALL_START` plus args; interrupt collector adds `tool_suspended` to final `RUN_FINISHED.outcome`.         | Both compress live suspended state; AG-UI carries richer interrupt metadata at run end.                                                                                                                |
| `tool_input_delta(toolCallId, delta)`              | `tool_call_update`, status `in_progress`, rawOutput/content text = delta.                      | Ensures `TOOL_CALL_START`, emits `TOOL_CALL_ARGS` delta.                                                        | Strong for streamed args in AG-UI; ACP treats args delta like output text because `ToolCallUpdate` has no args delta field.                                                                            |
| `tool_input_finished(toolCallId)`                  | None.                                                                                          | None.                                                                                                           | Gap. Tool argument completion is not visible at protocol level.                                                                                                                                        |
| `tool_updated(partialResult)`                      | `tool_call_update`, status `in_progress`, rawOutput and JSON preview.                          | `CUSTOM pea.tool.update { toolCallId, value }`.                                                                 | ACP preserves as normal tool update; AG-UI uses Pea custom event.                                                                                                                                      |
| `tool_shell_output(output, stream)`                | `tool_call_update`, status `in_progress`, rawOutput/content text.                              | `CUSTOM pea.tool.shell_output { toolCallId, value: { output, stream } }`.                                       | ACP could theoretically use terminal content but currently emits text. AG-UI needs custom event unless tool output becomes activity/state.                                                             |
| `tool_finished(result, isError, providerMetadata)` | `tool_call_update`, status `completed` / `failed`, rawOutput/content preview.                  | `TOOL_CALL_END`, then `TOOL_CALL_RESULT` with stringified result and role `tool`.                               | ACP preserves error status and raw output. AG-UI result lacks explicit error flag in core event. Provider metadata currently ACP-only in runtime event but not emitted as separate AG-UI custom event. |
| `plan_updated(tasks)`                              | `plan` entries, full replacement, `priority: medium`, status normalized.                       | `CUSTOM pea.plan.updated`.                                                                                      | ACP native; AG-UI custom.                                                                                                                                                                              |
| `plan_requested(title, plan)`                      | Markdown bullets converted to `plan` entries.                                                  | `CUSTOM pea.plan.requested`; interrupt collector adds `plan_approval_required`.                                 | ACP makes plan visible but approval itself is not represented by this mapper. AG-UI can expose interrupt at run finish.                                                                                |
| `runtime_error(source, error)`                     | None.                                                                                          | `CUSTOM pea.runtime.error`. Wrapper may also emit `RUN_ERROR` only if runtime send throws.                      | ACP error visibility gap. AG-UI has custom error telemetry plus wrapper-level run error.                                                                                                               |

## Mastra HarnessEvent Map

Legend:

- **Direct**: event maps into a Pea runtime event and then a protocol event/update.
- **Wrapper**: protocol wrapper emits equivalent lifecycle outside the `PeaRuntimeEvent` translator.
- **Custom**: protocol receives app-specific/custom event rather than native protocol event.
- **Dropped**: current adapter intentionally or accidentally emits nothing.
- **Gap**: data is useful but not represented today.

## Mastra v1 Harness Event Distinction

The installed `@mastra/core` package also exposes `dist/harness/v1/events.d.ts`, where `HarnessEvent` is a storage/event-emitter domain with:

| v1 event                    | Meaning                                                         | Current Pea protocol relevance                                                |
| --------------------------- | --------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `session_created`           | Persistent session created with resource/thread/mode/model ids. | Not the event union used by current Pea protocol translator.                  |
| `mode_changed`              | v1 mode change.                                                 | Similar name to broad union, but not imported by current Pea runtime adapter. |
| `model_changed`             | v1 model change.                                                | Same.                                                                         |
| `thread_cloned`             | v1 thread clone metadata.                                       | Same.                                                                         |
| Custom event `type: string` | JSON-serializable custom event.                                 | Same.                                                                         |

Audit consequence: do not assume every upstream `@mastra/core/harness` type named `HarnessEvent` is the same surface. Pea currently adapts the broad UI/runtime event union from `dist/harness/types.d.ts`.

## Adapter Depth Assessment

| Area                               | Depth                                                            | Evidence                                                                                                                                | Risk                                                                                                                           |
| ---------------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Assistant text                     | Deep enough for beta.                                            | Mastra message diff -> runtime start/delta/end -> ACP chunks / AG-UI text lifecycle.                                                    | ACP does not expose start/end; non-text message parts are mostly ignored.                                                      |
| Tool lifecycle                     | Deep for core tools.                                             | Start/input delta/update/shell/end all become runtime events; ACP gets tool cards; AG-UI gets native tool lifecycle plus custom output. | ACP reports streamed args as raw output; AG-UI uses custom for partial/shell/error metadata.                                   |
| Tool raw payloads                  | Strong.                                                          | `sanitizeJson` preserves JSON-safe full values; ACP `rawInput`/`rawOutput`; AG-UI stringifies args/results/custom payloads.             | AG-UI result is string content; structured consumers need custom payloads or parse JSON.                                       |

## Workbench Extension Metadata Decision

Current decision: Pea workbench-only data should ride ACP extensibility metadata, not new ACP top-level fields. Namespaced Mastra Harness / OM / debug / inspector / model / thread facts belong under ACP `_meta["pe.tools/workbench.update"]`, then project into `@pe/agent-projection` retained state. This keeps the ACP wire shape spec-compatible while preserving Pea-specific transparency for first-party UIs.

Use this path for data that is UI-relevant but not native ACP protocol state: raw workbench debug events, observational-memory lifecycle markers, system-prompt/context/raw-message inspector snapshots, model picker state, and retained thread/history summaries. Keep native ACP fields authoritative when ACP already has a concept, such as message chunks, tool calls, plans, permissions, session info, and current session mode.

Primary source paths for the current implementation:

- `source/pe-tools/packages/agent-contracts/src/index.ts`
- `source/pe-tools/packages/agent-projection/src/index.ts`
- `source/pe-tools/packages/acp-client/src/index.ts`

## Bottom Line

The current Pea adapters are already deep where beta auditability matters most: assistant text, tool lifecycle, raw tool input/result, shell output, plans/tasks, subagent inner tools, and AG-UI interrupt outcomes.

They are intentionally thinner around runtime/session telemetry, model/mode changes, token usage, OM progress, generic user questions, sandbox access requests, and ACP runtime errors. That means the adapters are not shallow wrappers, but they are also not comprehensive mirrors of Mastra Harness. They are focused adapters over a Pea-owned runtime event contract.

The clean next step is not to let ACP or AG-UI consume raw Mastra events. The clean next step is to widen `PeaRuntimeEvent` only for the facts Pea wants to make protocol-visible, then map those additions once per protocol.
