# Mental Model

Dev-agent context has two layers that must stay separate:

- the always-loaded harness layer: small routing rules, product boundaries, and proof invariants
- the skill layer: fat workflow programs for repeated Pe.Tools work

The harness exists to prevent harmful default behavior and dispatch to the right workflow. It should not duplicate the detailed loops that skills own.

# Source of Truth

The source surfaces are:

- `docs/features/dev-agent-context/_GOALS.md`: north star, non-goals, and durable philosophy
- `docs/features/dev-agent-context/_DEV.md`: implementation model and source-of-truth map
- `source/pea/app/dev-agent-instructions.ts`: thin always-loaded dev-agent router/invariant instructions
- `source/pea/app/dev-agent-skill-content/dev-agent-workflow-skills.ts`: dev-agent workflow skill source

Generated/runtime surfaces are:

- `.mastracode/skills/*/SKILL.md`: regenerated by dev-agent bootstrap from `dev-agent-workflow-skills.ts`; inspect as output only, do not hand-edit as source
- `source/pea/app/dist/*`: TypeScript build output; do not edit directly

`ensureDevAgentProjectFiles` rewrites repo-local `.mastracode/skills` during dev-agent bootstrap. A bounded verification pass can import the built `dist/dev-agent-project-files.js` and call `ensureDevAgentProjectFiles(process.cwd())` without starting the interactive `pea dev` TUI.

# Router Layer

`dev-agent-instructions.ts` should stay thin. It owns:

- Pea vs dev-agent product boundary
- proof-lane invariants for source compile, package/artifact, AttachedRrd, FreshRevitProcess, and installed-lane proof
- automatic skill activation from natural user language
- failure-mode routing to the right skill
- the Pe-native resolver substrate
- docs-first durability rules

It should not copy full skill loops. If the instructions begin accumulating a workflow, move that workflow into an existing skill or create a skill only after repeated use proves the need.

# Skill Layer

Dev-agent skills are project-scoped workflow programs. The most important skills are fat because they prevent frequent harmful defaults or coordinate many fragile micro-steps:

- `pe-steer`: ambiguous intent, "grill me", philosophy, vocabulary, boundaries, and durable capture
- `pe-live-loop`: Revit/Rider/RRD/hot reload/session freshness coordination
- `pe-diagnose`: broken behavior, regressions, confusing errors, and source-vs-product mismatch
- `pe-architecture`: product boundaries, module seams, interface shape, and deterministic-vs-latent judgment
- `pe-write-skill`: skill trigger reliability, skill design, and Pea/dev-agent skill separation

Descriptions are routing tables. They should include literal user phrases and failure-mode smells even when that is repetitive. Elegance matters less than reliable automatic activation.

Internal `Dispatch` or `Modes` sections are allowed, but they are for the agent's internal routing after activation. Do not require the user to know mode names. Explicit `/skill ...` is an override/debug affordance, not the primary UX.

# Context Resolution

Use Pe.Tools' existing documentation and generated contract model as the resolver substrate:

1. Read nearest `AGENTS.md` for durable agent behavior, cautions, and shared language.
2. Read relevant `_GOALS.md` for why, north-star intent, and non-goals.
3. Read relevant `_DEV.md` for mental model and architecture shape.
4. Read `docs/features/<feature>/` when the capability spans packages.
5. Inspect generated contracts/docs, host operations, schemas, and source when they are the current truth.
6. If durable truth is missing, treat that as signal and capture it in the nearest proper Pe doc.

Do not add Matt-style `CONTEXT.md`, `CONTEXT-MAP.md`, or ADR conventions in this model unless a future intentional design pass replaces the Pe-native taxonomy.

# Docs-First Rule

When a fat skill's durability checkpoint says durable capture is needed, docs happen before source implementation.

- In Build/default mode: update the nearest durable doc first, then implement.
- In Plan/read-only mode: make the durable doc update the first implementation step in the submitted plan and do not claim the capture already happened.
- If no durable capture is needed: state the reason briefly.

Durable capture should be focused. Capture reusable intent, boundaries, workflow rules, repeated failure modes, north-star changes, proof rules, and shared terms. Do not turn every task into doc churn.

# Pea Separation

Pea and dev-agent share bootstrap code but are separate products.

- Pea is the deployed Revit/operator workbench over `Pe.Host`, host operations, scripts, Revit API docs, logs, and bundled Pea workflow skills.
- Dev-agent is the MastraCode-based source-editing agent for this repo.
- Dev-agent may use Pea only as a black-box product feedback harness.
- Dev-agent repo tools, repo instructions, proof-lane posture, and repo workflow skills must not leak into installed Pea.

Pea implications from the thin-harness/fat-skills philosophy should stay in these dev-agent context docs until a Pea-specific design pass intentionally updates Pea operator skills.

# Future Promotion Rules

Do not add new surfaces preemptively.

- Add a standalone resolver skill only after embedded resolver sections become a proven cross-skill maintenance problem.
- Add skill scripts/assets only when deterministic repeated helper logic is safer as a reviewed file than as regenerated shell or prose.
- Add a bounded bootstrap CLI command only if repeated operator use needs it; current verification can call the exported function directly.
